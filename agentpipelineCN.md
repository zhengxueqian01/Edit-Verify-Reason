# Agent 总体流程

## 总览

ChartAgent 是一个先更新图表、再回答问题的 chart agent。

它的端到端行为可以概括为：

1. 读取统一的用户请求和图表。
2. 将请求拆成更新意图和 QA 意图。
3. 从当前 SVG 中恢复图表结构。
4. 规划需要执行的图表修改。
5. 在最新图表状态上逐 step 执行修改。
6. 检查更新后的渲染结果是否可用。
7. 基于更新后的图回答 QA 问题。
8. 如果问题视觉上较难，再增加一层轻量视觉推理。

当前系统是 hybrid 的：

- 模型负责理解、规划、回答和视觉工具选择
- 程序化图表流程负责 SVG grounding、图表编辑和结果校验

### Agent 的运行角色

这个 agent 不是单次直通式 pipeline，而是一个带重试的分阶段循环。

它在运行时主要做四类事情：

- 理解用户到底想改什么
- 理解当前图表长什么样
- 按请求顺序把图表改掉
- 在图表改完之后再回答问题

对于一些困难问题，它还会在第一次回答后追加一轮视觉推理。

## 输入理解和请求拆分

### 输入理解

agent 会把用户请求当作一条统一指令，而不是几条互不相关的输入通道。

这条统一请求里，往往同时混着两类信息：

- 自然语言的更新要求
- 更新完成后要回答的最终问题

在评测风格的输入里，这条请求后面还可能带有一段结构化尾部。出现这种情况时，agent 会把它当作同一条更新请求的更强 grounding，而不是另一份独立命令。

所以输入阶段并不只是“读文本”，而是在决定：

- 图表到底要改什么
- 更新完成后到底要回答什么
- 输入中是否带有应该在后续执行中被信任的显式结构化线索

### 请求拆分

第一个关键阶段，是把统一请求拆成两个语义部分：

- 更新侧
- QA 侧

更新侧应该只保留图表编辑意图。

QA 侧应该只保留“图表已经改完之后要回答的问题”。

这个阶段通常会先尝试模型拆分；如果效果不稳定，再退回规则拆分或启发式拆分。

它的核心目标不是生成一段漂亮的新文本，而是稳定地建立两个内部边界：

- 哪部分需要被执行
- 哪部分应该留到最后回答

如果原始请求里嵌了显式结构化信息，拆分阶段也会把这些信息保留下来，并继续传给后面的执行阶段作为 grounding。

但问题分解并不会停在这里。

系统先把统一请求拆成“更新侧”和“QA 侧”之后，还会继续把更新侧本身拆成一串有顺序的编辑步骤。

这是必要的，因为很多用户请求并不是单一步操作，而是会包含：

- 一句话里的多个操作
- 前后有依赖关系的连续操作
- 一个需要被拆成多个小动作的大型复合修改

所以这里要特别区分两个层次：

1. 请求拆分阶段只负责把统一请求稳定地拆成更新侧和 QA 侧
2. 真正把更新侧展开成可执行 step，是后面的更新规划和 step 构造阶段完成的

也就是说，请求拆分阶段可以识别“更新侧里可能包含多个动作”，但它本身不应该承担最终 step 序列生成的职责。

## 图表感知

### 图表感知

完成拆分后，agent 需要先理解自己要修改的图表。

这个 perception 阶段会从 SVG 中恢复图表结构，例如：

- 图表类型
- 坐标轴布局
- 图例或类别信息
- 图元分布
- 数据系列、点、线段、面积区域等 mark 的组织方式
- 颜色、标签、坐标和值之间的对应关系
- 后续执行修改所需的映射关系

这是整个系统的 grounding 阶段。它让 agent 能把用户请求和图表中的真实视觉元素对应起来。

更具体地说，这一阶段通常不是在“理解图片语义”这种宽泛层面停住，而是尽量把后续执行真正需要的结构恢复出来，例如：

- 哪些元素属于同一个 series
- 某个类别名在图中对应哪一组图元
- 坐标轴刻度和视觉位置之间怎样映射
- 当前图里哪些对象已经存在，哪些对象需要新增或删除

因此，图表感知的输出不只是一个描述性摘要，更像是后续 planner 和 updater 要依赖的内部工作上下文。

perception 也不是只做一次。

它会发生在：

- 初始规划前，对原始图先做一次
- 每个执行 step 开始前，对最新 SVG 再做一次

这种重复感知很重要，因为图表会不断变化，后续步骤必须基于已经更新过的图，而不是一直基于原图。

如果没有这一层 grounding，后面的模型即使理解了自然语言，也很难稳定知道“删的是哪条线”“改的是哪个类别”“新增内容应该挂到哪一组视觉结构上”。

所以图表感知在这个系统里既是理解阶段，也是执行前置阶段。

## 图表更新

### 更新规划

当图表结构被感知出来后，agent 会进入规划阶段。

这个阶段会把更新请求转换成一个有顺序的执行计划。

它的主要职责是：

- 把更新要求规范化成更清晰的内部意图
- 保留用户请求中的操作顺序
- 把多操作请求拆成可执行 step
- 让显式数据 payload 尽量跟正确的 step 对齐

如果用户请求里包含多个修改动作，planner 不应该把它们压成一个模糊动作，而应该尽量保留原本需要执行的先后顺序。

如果模型给出的规划不完整，系统也会退回到更偏规则的 step 构造方式。

从实际流程上看，这一阶段才是“把更新侧继续分解成步骤”的核心位置。

所以像“先删除一个类别、再新增一个类别、最后回答问题”这样的请求，在这里应该被展开成 step 序列，而不是继续保留成一整段笼统文本。

后面的执行阶段真正消费的，就是这串步骤。

如果只回答“step 到底在哪个环节拆出来”，答案应该是：

- 请求拆分负责分出 update 和 QA
- 更新规划负责把 update 解释成有顺序的编辑计划
- step 构造负责把计划最终落成执行器真正要跑的 step 列表

所以 `step 构造` 不是前面的 `Question Split` 本身。

更准确地说：

- 从语义上，它确实延续了前面“把复杂请求拆开”的思路
- 但从流程位置上，它属于图表更新环节内部，是执行前的准备步骤
- 执行器真正消费的 step 列表，就是在这里最终定稿的

在当前实现里，还要再补一个实现细节：

- planner 本身已经会产出一版 `steps`
- 如果 `SVG_UPDATE_MODE=llm`，代码里会把它规范化成 `llm_intent`
- 在这种模式下，系统会额外再跑一次基于 perception 的 step 重写逻辑

所以从代码角度说，“step 在哪一层产生”并不是只有一次回答：

- 第一次是在 planner
- 第二次是可选的 `llm_intent` step rewrite
- 最后 `step 构造` 再负责补齐、合并和展开原子步骤

### 可选的 Step Rewrite（代码里叫 llm_intent）

更新流程中还可以插入额外一层模型阶段。它在文档语义上可以理解成“意图细化”，但按当前代码实现，更准确的名字其实是可选的 step rewrite。

这一层不会直接改图。它会在 planner 之后运行，并利用：

- 已规范化的更新请求
- 图表感知摘要
- 之前已经提取到的结构化 grounding

再生成一版 step 级描述；如果成功，就会覆盖 planner 原先给出的 `steps`。

它的触发条件不是“所有请求都自动走”，而是由更新模式控制：

- 如果 `SVG_UPDATE_MODE=rules`，这层不会启用
- 如果 `SVG_UPDATE_MODE=llm`，内部会规范化成 `llm_intent`
- 这时系统会额外调用一次 LLM，根据 operation text、图表感知摘要和结构化上下文重写 `steps`

所以这里的 `llm_intent` 不是一整套新的 update pipeline，它只影响“step 怎么写、要不要被重写”这一小段。

可以把它理解成：

- planner 已经知道“大概要做哪几步”
- 但当前实现里，这一层不只是补充字段，而是可能直接重写整组 `steps`
- 重写时会参考 perception 和 structured grounding，让步骤更贴近当前图表

例如 planner 可能只产出“删除一个不需要的 series”，而这一层会进一步改写成更接近执行的描述，比如“删除图例中名称为 X、颜色为 Y 的那条 series”，前提是这些信息已经能从 perception 和 grounding 中拿到。

再比如，一条复合请求原本可能是：

- 把 A 系列 2020 年和 2021 年的值都改掉
- 删除 B 和 C 两个类别

那么这层和后续 step 构造组合起来后，最终很可能不会保留成两条模糊指令，而会继续展开成更细的 step，例如：

1. Change "A" in 2020 to ...
2. Change "A" in 2021 to ...
3. Delete the category/series "B"
4. Delete the category/series "C"

也就是说，“多个年份”或“多个目标”的更新，在当前实现里通常会被拆成多个原子 step，而不是让执行器一次吃下一整段复合描述。

这一点尤其体现在后面的 step 构造阶段：

- 一个 `change` 如果同时包含多个年份，会按年份拆开
- 一个 `delete` 如果同时包含多个 category，也会按目标拆开
- 一个复合请求如果同时包含 add / delete / change，会尽量保持原请求顺序展开

所以它的作用不是替代现有的编辑流程，而是把 planner 产出的步骤进一步改写到更利于执行的形式。

所以即使走了更 model-driven 的路径，整体分工仍然不变：

- 模型负责进一步说清楚“要改什么”
- chart-specific executor 负责真正把 SVG 改掉

这一步是可选层，不是所有请求都必须经过它。如果当前模式不是 `llm_intent`，或者这一步没有成功产出有效 `steps`，系统就会继续使用 planner 原有的结果进入 step 构造和执行。

### Step 构造

在正式执行前，agent 会先把最终要运行的 step 列表定下来。

这个过程会综合多种信息来源：

- planner 生成的编辑步骤
- 结构化更新线索
- 原始更新文本中的 fallback 解释

当遇到复合请求时，这一层还会把大动作拆成更原子的可执行动作。

比如：

- 一个请求里包含多个删除
- 一个请求里包含多个独立的数值改动

这些都可能被展开成顺序明确的小 step。

它的目的，是让执行面对的是一列清晰、可落地的动作，而不是一条大而模糊的总命令。

所以从语义意义上说，这一层并不是和“问题分解”完全分开的。它是前面 update 分解思路在执行视角下的延续：

- 先把统一请求拆成更新和 QA
- 再把更新计划整理成执行器真正可运行的步骤列表

### 逐 Step 执行

执行阶段是迭代式的。

对于每一个 step，agent 会：

1. 把该 step 转成更具体的执行指令
2. 对当前图表状态重新做 perception
3. 调用对应图表类型的 updater
4. 产出新的 SVG 和渲染结果

虽然不同图表类型会走不同 updater，但整体模式一致：

- 先结合最新图表结构理解当前 step
- 再修改 SVG
- 然后保存结果

这意味着多步请求是在一个不断演化的图表上执行的。

也就是说：

- 第一步改的是原始图
- 第二步改的是第一步产出的图
- 第三步改的是第二步产出的图

这就是当前 chart update agent 的核心循环。

### 校验与重试

一次执行 attempt 完成后，agent 会检查更新结果是否足够可信，能不能进入 QA。

这个校验阶段有两个目标：

- 确认渲染结果本身仍然有效
- 确认图表修改很可能以图表一致的方式成功落地

系统会同时使用通用渲染检查和部分 chart-specific 的程序化检查。

如果校验失败，agent 不会立刻放弃，而是会带着前一次失败得到的提示，重新走一轮 perceive-plan-execute 主循环。

所以更新阶段不是简单的：

- 规划一次
- 执行一次

而更像：

- 规划
- 执行
- 校验
- 失败则重试

这种 retry 结构，是它更像 agent 而不是一次性 pipeline 的重要原因。

## 更新后回答与可选视觉增强

### 基于更新后图表回答

当更新后的图被接受后，agent 才会正式回答 QA 问题。

这一阶段回答的是已经拆出来的纯 QA 问题，而不是原始混合请求。

这里的关键点是：

- 更新要求不应该在回答阶段再次被当作待执行命令
- QA 阶段此时应该退化成纯粹的 chart question answering

因此最终回答依赖的是更新后的视觉结果，以及一份对更新过程的压缩摘要。

### Visual Tool Augmentation

有些图表问题即使在更新完成后，纯视觉上仍然很难直接推理。

典型例子包括：

- cluster 推理
- intersection 计数
- 极值点定位

对于这种情况，agent 会在第一次回答后额外运行一层 visual tool 阶段。

这一层不会重新改图表内容，而是添加一些轻量视觉引导，例如：

- 高亮区域
- 标记点
- 绘制辅助线
- 隔离相关拓扑

它的目的不是展示效果，而是提升困难视觉任务上的回答可靠性。

如果这一步成功，系统会在增强后的图上再回答一次，并把这个结果作为最终答案。

## 为什么这是 Agent

当前系统更适合被称为 agent，而不是普通 pipeline，因为它具备：

- 把一条请求分解成多个语义任务
- 基于当前图表状态做 grounding
- 按顺序执行多步修改
- 状态变化后重新感知
- 回答前先校验
- 失败后重试
- 困难问题时使用额外工具

所以它不是简单的“输入问题，输出答案”。

它是一个围绕不断演化图表状态运行的 update-and-reason loop。

## 总结

一句话概括：

ChartAgent 会先把编辑意图和 QA 意图拆开，再把编辑请求 grounding 到当前图表上，经过重复感知、逐步执行和校验后，在更新后的图表上回答问题，并在问题困难时加入一层轻量视觉推理。
